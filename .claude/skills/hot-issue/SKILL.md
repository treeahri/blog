# 핫이슈 카드뉴스 작성 워크플로우

목표: 실제로 화제성 있는 경제·생활 주제(정부 지원금, 공공요금, 할인·환급 제도, 여행/휴가철
절약 팁 등)를 리서치해서 `blog-image` 저장소의 `hot-issue/current.json`에 카드뉴스 데이터로
커밋한다. 사진이 필요한 슬라이드(`imageText`/`imageCompare`)는 `imageDataUrl`을 비워두고
notes에 남긴다 — 실제 사진 첨부는 앱에서 사람이 한다.

## 0. 설정 확인

원격(Claude Code on the web) 세션에서는 GitHub MCP 도구로 직접 커밋한다. 로컬 세션이라면
`naver-blog-review` 스킬과 동일한 `config.local.json`(`{"owner", "repo", "token"}`, 저장소는
`blog-image`)을 공유해서 스크립트로 읽고 쓸 수 있다.

## 1. 이전 이슈 확인 (중복 방지 & 예고 이어가기)

`hot-issue/archive/`를 확인해서 최근에 어떤 주제를 다뤘는지 파악한다.

- 같은 주제를 반복하지 않는다.
- 직전 글의 `body.closing`에 "다음엔 ~로 찾아올게요" 같은 예고가 있으면 자연스럽게 그
  주제로 이어간다. 다만 그 시점에 더 화제성 있는 주제가 있으면 억지로 끼워 맞추지 않고
  그쪽을 우선해도 된다.

## 2. 주제 선정

- 카테고리: 경제 또는 생활 (`meta.category`)
- 니치: 정부 지원금·환급, 세금·보험 청구 절차, 공공요금·복지할인, 여행/휴가철 절약,
  간편결제·카드 혜택 등 "아는 사람만 챙기는 돈 되는 생활 정보".
- WebSearch로 최근 실제로 화제이거나 지금 시행 중인 정책·제도를 찾는다 — 지어내지 않는다.

## 3. 리서치

- WebSearch/WebFetch로 조사한다. 수치(%, 원, kWh, 기한 등)는 **최소 2개 이상의 독립된
  출처**에서 일치하는지 교차 확인한다. 한 곳에서만 나온 수치는 신뢰도가 낮으니 다른
  표현으로 재검색하거나, 그래도 확인이 안 되면 `notes`에 명시하고 표현을 완곡하게 낮춘다.
- WebFetch가 403 등으로 막히면 WebSearch 결과 요약을 근거로 쓰되, `sources`에는 실제
  검색된 원문 URL을 넣는다.
- **대상자·그룹별로 조건이 갈리는 제도는 뭉뚱그리지 않는다.** 예를 들어 전기요금
  복지할인처럼 "다자녀·대가족은 정률 할인, 장애인·기초수급자는 정액 할인"인 식으로
  계산 방식 자체가 다른 경우, 하나의 수치로 합쳐 쓰면 오류가 된다 — 그룹마다 따로 적는다.

## 4. 톤·문체 규칙

`naver-blog-review` 스킬의 3-1절과 동일한 원칙을 따른다:

- 페르소나: "다리뷰다"(daReviewDa). 해요체 위주, 합니다체 섞어서.
- 문장은 짧게 끊는다. AI 티나는 상투구("요즘 많은 분들이", "~에 대해 알아보겠습니다")와
  단정적 광고 문구("무조건", "완벽한", "확실히 효과") 금지.
- 근거 없는 수치·효과 단정 금지. 완곡 표현("~할 수 있어요", "~인 편이에요")을 쓴다.

## 5. 스키마 매핑 (`src/hot-issue/types.ts`의 `HotIssueData`)

- `slides`: `cover`(표지) → `quote`/`table`/`checklist`/`imageText`/`imageCompare` 등을
  조합(보통 4~6장) → `outro`(마무리).
- `imageText`/`imageCompare`는 실제 사진이 없으면 `imageDataUrl`을 빈 문자열로 두고,
  `notes`에 "실제 사진은 앱의 핫이슈 모드에서 직접 첨부 필요"라고 남긴다.
- `body`: `postTitle` / `greeting`("안녕하세요, 다리뷰다입니다.") / `intro` / `closing` /
  `disclaimer` / `hashtags`.
- `sources`: 실제 리서치에 쓴 링크만 `name` + `url`로.
- `notes`: 확인이 더 필요한 부분, 시점에 따라 달라질 수 있는 정보.

## 6. 파일 저장

- `hot-issue/current.json` 갱신 (앱이 바로 읽는 파일).
- `hot-issue/archive/YYYYMMDD-<slug>.json`에 동일 내용을 별도 저장 (영구 기록, 이후에도
  덮어쓰지 않음).
- `slug`는 영문 kebab-case로 주제를 짧게 표현.

## 7. 최종 확인 (커밋 전 필수, 생략 금지) ⚠️

커밋하기 직전, 반드시 아래 두 가지를 **순서대로** 확인한다.

1. **팩트체크 재검증** — 작성한 문장 속 수치·기한·조건을 하나씩 다시 훑으며, 3절에서 확인한
   출처와 실제로 일치하는지 재대조한다(가능하면 출처를 바꿔 한 번 더 교차 검색). 특히
   아래 유형의 실수가 없는지 확인한다:
   - 대상자·그룹마다 조건이 다른 제도를 하나로 뭉뚱그려 쓰지 않았는지 (예: "장애인도
     다자녀와 똑같이 30% 할인" — 실제로는 정액 할인일 수 있음).
   - 완화된 조건(예: "1%만 아껴도")과 원래 조건(예: "기존엔 3%")을 헷갈려 쓰지 않았는지.
   - 기간 한정 조치(예: "7~8월만 한시")를 상시 제도처럼 쓰지 않았는지.
   - 출처에 없는 숫자를 임의로 만들어내지 않았는지.

   불일치나 근거 부족을 발견하면 그 문장을 고치거나 완곡한 표현으로 낮추고, `notes`에 왜
   낮췄는지 남긴다.

2. **오탈자·맞춤법 확인** — `slides`의 모든 `title`/`body`/`items`/`columns`/`rows`와
   `body`의 모든 필드(`postTitle`/`greeting`/`intro`/`closing`/`disclaimer`/`hashtags`)를
   처음부터 끝까지 다시 읽으며 오탈자, 띄어쓰기 오류, 조사 오류(을/를, 이/가 등), 문장
   중복·누락을 확인한다. 리서치 내용을 붙여넣거나 요약하며 생긴 어색한 문장은 자연스럽게
   다듬는다.

이 두 확인을 마치기 전에는 커밋하지 않는다. 문제를 고쳤다면 JSON을 다시 검증(예:
`python3 -m json.tool`)한 뒤 커밋한다.

## 8. 안내

1. 앱에서 "저장소에서 불러오기" 클릭.
2. 미리보기 확인, 사진이 필요한 슬라이드(`imageText`/`imageCompare`)는 입력 탭에서 직접
   첨부.
3. 발행.

## 주의

- 근거 없는 수치·효과 단정 금지 (3절·7-1절 참고).
- 제도별 대상자 조건을 단순화해서 왜곡하지 않는다 (3절·7-1절 참고).
- `config.local.json`의 토큰을 출력하거나 커밋하지 않는다.
