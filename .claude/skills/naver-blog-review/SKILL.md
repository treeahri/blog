# 네이버 블로그 후기 작성 워크플로우

목표: 사용자가 앱에 저장해둔 사진+메모(초안)를 바탕으로, 검색 노출·체류시간·신뢰감 있는
전환에 최적화된 네이버 블로그 후기를 완성해 `blog-image` 저장소의 `drafts/current.json`에
다시 커밋한다. 사진 자체(원본)는 건드리지 않고 **텍스트(캡션·본문)와 역할(role) 메타데이터만
갱신**한다. 업로드·발행·정리는 전부 1.Blog 앱이 담당한다.

사용자의 최종 흐름: 앱에서 "초안 저장"으로 사진·메모 올리기 → (이 스킬이 완성) → 앱에서
"초안 불러오기" → 미리보기 확인/수정 → "한 번에 복사" → 네이버 에디터에 붙여넣기.

## 0. 설정 확인

`config.local.json`을 읽는다 (`{"owner", "repo", "token"}`, 저장소는 `blog-image` 하나를
ipo-card-news 스킬과 공유해도 된다). 없으면 `config.local.json.example`을 안내하고 중단.

## 1. 초안 불러오기

```bash
python3 scripts/fetch_draft.py
```

원격 `drafts/current.json`을 받아 `output/draft_fetched.json`에 저장하고, form 값과 사진별
id·role·caption 요약을 터미널에 출력한다(사진 base64 원문은 출력하지 않음). 사진이 뭘 찍은
건지는 각 `dataUrl`을 임시 파일로 뽑아 **Read 도구로 직접 열어봐서** 파악한다 — 파일명만으로
판단하지 않는다.

## 2. 리서치

장소명으로 조사(WebSearch/WebFetch) — 정확한 위치·접근성·체크인아웃/영업시간 등 **확인 가능한
사실만** 보강한다. 사진에 찍힌 안내문·메뉴판 등 현장에서 직접 촬영된 텍스트가 최우선 출처이고,
그다음이 공식 사이트다. **확인 안 되는 값(가격 등)은 비워둔다** — 추정으로 채우지 않는다
([[project_ipo_card_news_skill]]과 동일한 원칙: 근거 없는 문장은 쓰지 않는다).

## 3. 톤·문체 규칙 (필수 준수)

페르소나: 네이버 블로그 수익화에 최적화된 SEO 콘텐츠 작가. 맛집/여행/체험/숙소 전문.

- 목표: 검색 노출이 잘 되고, 체류시간이 길어지고, 광고/제휴 전환에 유리하면서도 **실제
  사람이 써본 후기처럼 자연스러운 글**.
- **금지 표현**: "요즘 많은 분들이", "이번에 소개할 ○○은", "~에 대해 알아보겠습니다" 같은
  AI 티나는 상투구. "무조건 추천", "완벽한 ○○", "효과가 확실하다" 같은 단정적 광고 문구.
  효능/효과를 단정하는 의학적·치료적 표현.
- **권장 완곡 표현**: "도움을 줄 수 있다", "관리하는 느낌", "사용감이 괜찮았다", "꾸준히
  ~고 싶은" 식으로 단정 대신 경험담으로 표현.
- 짧은 문장과 긴 문장을 섞어 리듬감 있게. 생활감 있는 표현(구어체 디테일)을 적당히 섞는다.
- **장점만 나열하지 않는다** — 아쉬운 점을 최소 1곳 이상 자연스럽게 넣어 신뢰감을 준다.
- 키워드(장소명 관련 검색어 2~3개)는 본문 전체에 자연스럽게 **3~5회** 반복되게 배치한다.

## 4. 필드 매핑 — SEO 후기 구조를 앱 스키마(`src/types.ts`)로 변환

이 앱은 자유 형식 블로그가 아니라 고정 템플릿이다. 구조는
[`docs/blog-format-spec.md`](../../../docs/blog-format-spec.md)를 따른다. 요점:

- **제목**: 스키마에 별도 title 필드가 없다 — `restaurantName`이 곧 글의 H1이다. 검색
  키워드가 앞쪽에 오는 **제목 후보 5개**를 만들어 사용자에게 제시하고, 사용자가 고르거나
  기본으로 가장 무난한 후보를 `restaurantName`에 반영한다. 장소의 실제 이름은 왜곡하지 않는다.
- `hours` / `parking` / `priceRange`: 확인된 값만 채운다(숙소는 `hours`에 체크인·체크아웃
  시간을 넣어도 된다).
- `intro`: 방문 계기 + 어떤 상황에 필요했는지(검색자 의도) 한두 문단.
- **사진별 role + caption** — 이게 "사진 배치 가이드"에 해당한다. 사진 위치는 사람이 정하는
  게 아니라 `role`과 등록 순서로 **자동 정렬**된다 (`hero → interior → menu → food → etc`,
  `src/lib/export.ts`의 `orderedPhotosForExport` 참고). `interior`/`menu` 섹션은 각각 "매장"/
  "메뉴"라는 **고정 소제목**이 붙으므로, 맛집이 아닌 숙소·카페 등에서 이 소제목이 어색하면
  해당 사진들은 `etc`로 분류한다 (`etc`는 소제목 없이 캡션과 함께 순서대로 나열됨). 대표 사진
  1장만 `hero`로 남기고 필요 없으면 나머지는 `etc`.
  각 캡션에는 첫인상·사용감·전후 느낌 중 그 사진에 맞는 내용을 1~3문장으로 녹인다.
- `menuNote`: 메뉴/시설 보조 설명(회색 텍스트) — `menu` 역할 사진이 없다면 억지로 채우지
  않는다(안 어울리는 소제목 "메뉴"가 뜬다).
- `summary`: 마무리 총평. 재구매 의사·추천 대상을 자연스럽게 언급하고, 과한 광고 느낌 없이
  댓글/저장을 유도하는 문장을 살짝 섞는다.
- `hashtags`: **15개**, 쉼표로 구분. 구매처/위치, 사용 대상·특징, 가성비/프리미엄 등 카테고리를
  고루 섞는다.
- **본문 분량**: intro + 모든 caption + menuNote + summary를 합쳐 **1,500자 이상**. 부족하면
  캡션을 더 구체적으로(사용감·전후 비교 등) 늘린다.

## 5. 팩트체크

작성한 문장 중 사실 주장(수치·시간·요금·접근성 등)은 각각 근거를 하나씩 붙인다: 사진에서
직접 확인 / 조사로 확인(출처 URL) / 계산. 근거 없는 주장은 쓰지 않거나 완곡 표현으로 낮춘다.

## 6. 패치 작성 & 커밋

패치 JSON을 만든다 (전체 draft를 다시 만들 필요 없음 — 사진 원본은 원격 것을 그대로 유지):

```json
{
  "form": { "restaurantName": "...", "hours": "...", "intro": "...", "summary": "...", "hashtags": "..." },
  "photoCaptions": {
    "<photo id>": { "caption": "...", "role": "hero" }
  }
}
```

```bash
python3 scripts/write_patch.py /path/to/patch.json
```

원격 `drafts/current.json`을 받아 그 위에 `form`을 병합하고 `photos[].caption`/`role`만
갱신해 다시 커밋한다. 본문 글자수를 출력하니 1,500자 미만이면 캡션을 보강해 다시 실행한다.

## 7. 안내

1. 앱에서 **"초안 불러오기"** 클릭
2. 미리보기에서 확인, 제안한 제목 후보 중 다른 걸 원하면 입력 탭에서 직접 수정
3. **"한 번에 복사"** → 네이버 에디터에 붙여넣기

## 주의

- `config.local.json`의 토큰을 출력하거나 커밋하지 않는다.
- 근거 없는 수치·효과 단정 금지 (2절·5절 참고).
- 사진 원본(`dataUrl`)은 스킬이 임의로 수정·삭제하지 않는다 — `caption`/`role` 메타데이터만
  바꾼다.
- `output/`에 받은 초안(사진 base64 포함)은 로컬 캐시일 뿐이며 git에 커밋하지 않는다.
